# DDPM 网页小游戏方案

## 项目概念

项目名建议：`噪声考古学家`。

一句话描述：玩家在一张不断去噪的图像里抢先识别隐藏目标，并通过有限的干预手段影响 DDPM 的生成方向，从而在“看懂图像”和“操纵图像”之间做取舍。

这个方案的重点不是把 DDPM 当成纯美术工具，而是直接把“从噪声到图像”的过程做成核心玩法。

## 目标

首版目标是做出一个能在网页里完整游玩的 MVP，验证两件事：

1. 玩家是否会对“逐步显形”的识别过程上瘾。
2. DDPM 的中间过程是否能稳定支撑可读、可猜、可干预的玩法。

MVP 不追求大模型、不追求高分辨率、不追求开放文本生成，优先验证玩法闭环。

## 核心玩法循环

单局流程：

1. 系统随机抽取一个目标类别，例如“猫”“潜水艇”“城堡”“机器人”。
2. 后端生成一张初始噪声图，并开始按时间片逐步去噪。
3. 前端每隔一小段时间显示新的中间帧，玩家观察图像逐渐成型。
4. 玩家在过程中做有限干预：
   - 提前猜答案
   - 使用引导卡改变生成风格
   - 冻结局部区域，强行保留当前画面
5. 图像完全成型或玩家猜中后结算得分。
6. 进入下一关，类别相似度更高、干预次数更少、时间更紧。

玩家感受到的核心张力：

- 猜早了，可能错失机会
- 猜晚了，分数变低
- 干预多了，更容易看懂，但奖励下降
- 干预错了，图会朝错误方向发展

## 首版规则设计

建议先做离散类别猜谜，不做自由文本输入判题。

### 类别池

- 初版类别数量：`24 ~ 40`
- 类别结构：
  - 动物
  - 交通工具
  - 建筑
  - 日常物体
  - 幻想生物

建议优先挑视觉特征强、轮廓明显的类别，避免“杯子”和“花瓶”这种过近类别一开始就劝退玩家。

### 局内资源

- 猜测次数：`3`
- 引导卡次数：`2`
- 冻结区域次数：`1`
- 单局时长：`20 ~ 30 秒`
- 去噪展示帧数：`20 ~ 40 帧`

### 计分公式

可以先用简单公式：

`总分 = 基础通关分 + 剩余时间分 + 提前猜中奖励 - 错误猜测惩罚 - 干预消耗惩罚`

建议参数：

- 基础通关分：`100`
- 每秒剩余时间：`+5`
- 在 50% 进度前猜中：`+50`
- 每次猜错：`-20`
- 每张引导卡：`-10`
- 冻结区域：`-15`

这样能把“少干预、高判断”自然导向高分打法。

## 可交互机制

### 1. 猜答案

这是主操作。首版建议做成“从候选词中选择”而不是自由输入：

- 减少 NLP 判题成本
- 避免同义词匹配问题
- 让 UI 更清晰
- 更适合快节奏网页体验

建议显示 `6` 个候选项，其中只有 `1` 个正确答案，其余为干扰项。

### 2. 引导卡

引导卡不是改变目标类别，而是改变生成偏向，让图像更容易被识别或更混乱。

首版可做的卡片：

- `锐化轮廓`：让边缘更明显
- `机械风`：偏金属结构
- `生物风`：偏柔软曲线
- `像素稳定`：减少抖动和形变
- `错误引导`：高风险高回报，若与目标匹配则更清晰，不匹配则更怪

实现上不需要真的让模型理解复杂自然语言，首版可以把引导卡映射成固定条件向量或固定分支轨迹。

### 3. 冻结区域

玩家框选当前图像中的一块区域，后续步骤尽量保留这块内容。

玩法意义：

- 玩家觉得“这个轮廓像头部”时可以锁住
- 允许产生“赌一把”的策略
- 能放大 DDPM 局部修补与逐步重构的特性

技术上可以先做弱版本：

- 不做真实扩散 mask 编辑
- 只是在预生成轨迹里切到“保留局部细节更久”的分支

## 难度曲线

首版建议做 3 个章节，每章 8 关。

### 第一章：新手识别

- 类别差异大
- 去噪速度慢
- 候选词少
- 引导卡效果明显

### 第二章：相近干扰

- 加入相似目标
- 去噪速度变快
- 错误引导更常出现
- 候选项更多

### 第三章：混沌遗迹

- 类别更抽象
- 初始噪声更强
- 可用干预次数更少
- 一些关卡会带风格偏移

## 美术与表现方向

不建议第一版追求写实。低分辨率和中间帧形变更适合风格化包装。

推荐方向：`考古扫描仪 + 像素噪声终端`

视觉关键词：

- 老式科研设备界面
- 琥珀、深青、暗绿配色
- 噪点、扫描线、网格背景
- 逐步刷新的屏幕感
- 低分辨率目标图像

这样有三个好处：

1. 低清图反而合理。
2. DDPM 的“模糊到清晰”更像仪器解译过程。
3. UI 能自然承载分数、提示、扫描参数等信息。

## 网页界面草图

建议单页完成主要体验。

### 布局

- 顶部：关卡、剩余时间、总分、连胜
- 左侧：本局日志与提示
- 中间：主画布，显示当前去噪图像
- 右侧：引导卡、冻结工具、剩余资源
- 底部：候选答案区与确认按钮

### 文本草图

```text
+-----------------------------------------------------------+
| Chapter 1-3        Time 18s        Score 1280     Combo 2 |
+----------------------+----------------------+-------------+
| Signal Log           |      Main Canvas     | Tool Rack   |
| - residual high      |                      | [锐化轮廓]  |
| - shape emerging     |      64 x 64         | [生物风]    |
| - class: animal      |    denoise frame     | [冻结区域]  |
|                      |                      | uses 2/3    |
+----------------------+----------------------+-------------+
| Candidates: [猫] [狐狸] [狼] [机器人狗] [城堡] [潜水艇]     |
|                     [Confirm Guess]                         |
+-----------------------------------------------------------+
```

### 交互节奏

- 去噪动画每 `500ms` 推进一次
- 玩家点击答案立即判定
- 用卡后主画布出现一次明显跳变，强化“干预生效”的反馈
- 猜中后播放快速复原动画和结算面板

## MVP 功能清单

### 必做

- 单人闯关
- 类别随机抽取
- 去噪帧序列播放
- 候选答案猜测
- 分数结算
- 三种基础引导卡
- 一次冻结区域
- 24 关内容配置
- 基础音效和转场

### 可延期

- 排行榜
- 成就系统
- 每日挑战
- 玩家对战
- 自定义 prompt
- 更高分辨率模型

## 技术选型建议

## 前端

- 框架：`React + Vite`
- 绘制：`Canvas 2D`
- 状态管理：`Zustand` 或 React 本地状态
- 动画：CSS + `requestAnimationFrame`
- 通信：首版可用 HTTP 轮询，后续再上 `WebSocket`

原因：

- 这类游戏 UI 较轻，React 足够
- Canvas 适合频繁刷帧
- Vite 启动快，适合快速验证玩法

## 后端

- 框架：`FastAPI`
- 模型：`PyTorch`
- 图像处理：`Pillow` / `NumPy`
- 会话存储：内存字典即可，后续再换 Redis

原因：

- FastAPI 很适合快速提供推理接口
- Python 侧更容易接 DDPM 训练和推理代码

## 部署

首版建议拆成两个服务：

- 前端静态站点
- 后端推理服务

如果只是本地 demo，也可以直接本机启动：

- 前端：`localhost:5173`
- 后端：`localhost:8000`

## 数据流设计

### 开局

1. 前端请求 `/session/start`
2. 后端返回：
   - `session_id`
   - `chapter`
   - `candidate_labels`
   - `initial_frame`
   - `remaining_tools`

### 推进

1. 前端每隔一段时间请求 `/session/{id}/step`
2. 后端返回：
   - `frame_index`
   - `image`
   - `progress`
   - `status`

### 玩家操作

前端调用：

- `/session/{id}/guess`
- `/session/{id}/use-card`
- `/session/{id}/freeze`

### 结算

后端返回：

- 是否猜中
- 最终目标
- 分数明细
- 下一关解锁状态

## DDPM 实现路线

这里建议严格分阶段，不要第一天就追求“浏览器里实时跑真正的 DDPM”。

### 路线 A：伪实时 MVP

这是最推荐的首版。

做法：

1. 后端提前为每个目标类别生成若干条去噪轨迹。
2. 每条轨迹保存 `20 ~ 40` 张关键帧。
3. 开局时随机抽一条轨迹播放。
4. 玩家使用引导卡或冻结区域后，切换到另一条更匹配的轨迹分支。

优点：

- 开发快
- 稳定性高
- 对显存和时延要求低
- 容易把玩法先做对

缺点：

- 严格来说不是实时连续扩散
- 玩家干预自由度有限

但这很适合验证游戏是否成立。

### 路线 B：小型真实 DDPM

当玩法通过后，再替换成真实推理。

建议参数：

- 图像尺寸：`32x32` 或 `64x64`
- 类别数量：`10 ~ 20`
- 条件输入：类别嵌入 + 少量风格卡
- 采样步数：尽量压缩到 `10 ~ 20`

可以考虑：

- DDPM / DDIM
- Latent Diffusion 的极小版本
- 直接使用已有小模型做微调

真实版本的关键不是追求画质，而是保证：

1. 中间帧可读
2. 帧间变化平滑
3. 每局推理延迟可接受

## DDPM 简化方案

如果是个人项目或小团队，建议按下面的难度顺序推进。

### 方案 1：预生成序列

- 直接离线生成所有中间帧
- 游戏运行时只做播放与分支切换

适用阶段：玩法验证

### 方案 2：固定类别的小模型

- 训练一个只认识少量类别的条件扩散模型
- 不支持自由输入 prompt
- 引导卡只映射到少量固定条件

适用阶段：技术验证

### 方案 3：局部可编辑扩散

- 增加 mask 条件
- 允许冻结/修改局部区域
- 让干预真正改变后续生成

适用阶段：正式版核心技术升级

## 训练和数据建议

如果要自己做小模型，先别碰大而杂的数据集。

建议做一个小型受控数据集：

- 每类 `500 ~ 2000` 张图
- 统一背景或纯色背景
- 尽量固定视角
- 先转成像素风或简化图标风

可选数据来源方向：

- 开源 icon 数据集
- 简笔插画数据
- 自建合成图

这样中间帧更稳定，也更容易被玩家识别。

## 服务器性能预估

首版按伪实时算，压力很低。

### 伪实时模式

- 后端只返回静态图片帧
- 可以缓存大量结果
- 普通云主机就能跑

### 真实小模型模式

- 需要 GPU 才能把时延压到可玩范围
- 单局最好控制在 `1 ~ 3 秒` 的总推理预算内
- 若逐帧在线推送，必须严格控制采样步数

## 两周 MVP 开发排期

### 第 1 周

1. 搭前端单页框架
2. 完成主画布、候选答案区、计分区
3. 用假数据播放去噪帧序列
4. 接入开局、推进、猜测结算
5. 配好基础 UI 风格和动效

### 第 2 周

1. 加入引导卡与冻结区域
2. 做关卡配置系统
3. 补音效与结算面板
4. 用离线生成的真实 DDPM 中间帧替换假图
5. 调整节奏、分数、难度

两周结束时应该达到：

- 可连续游玩
- 画面统一
- 基本有复玩价值
- 已经能测试玩家是否喜欢这个核心玩法

## 风险与规避

### 风险 1：模型太慢

规避方式：先用预生成轨迹，不把玩法验证绑死在实时推理上。

### 风险 2：中间帧不可读

规避方式：选轮廓强的类别，并控制数据风格统一。

### 风险 3：玩家理解不了干预价值

规避方式：每次用卡必须让画面产生明显反馈，不要做隐性效果。

### 风险 4：候选答案太简单或太难

规避方式：把类别配置做成表驱动，方便快速迭代。

## 建议的下一步

如果立刻开工，建议按这个顺序执行：

1. 先做纯前端假数据原型，验证节奏和 UI。
2. 再补一个最小 FastAPI 服务，提供关卡和帧序列接口。
3. 然后导入离线生成的 DDPM 中间帧。
4. 最后再决定是否投入训练一个真实可交互的小模型。

## 首版一句话定位

`一款把扩散模型去噪过程直接做成玩法的网页识别闯关游戏。`
